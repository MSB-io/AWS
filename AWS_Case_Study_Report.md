# FanEngage Sports Fan Platform - AWS Case Study Report

## 1. Cloud Architecture

The platform follows a scalable, decoupled architecture on AWS, using EC2 for compute, RDS for data, and S3 for backups.

```mermaid
architecture-beta
    group aws(cloud)[AWS Cloud]
    
    group vpc(cloud)[VPC: 10.0.0.0/16] in aws
    group public_subnet(cloud)[Public Subnet: 10.0.1.0/24] in vpc
    group private_subnet(cloud)[Private Subnet: 10.0.2.0/24] in vpc
    
    service igw(internet)[Internet Gateway] in vpc
    
    service ec2(server)[EC2 Instance\nFrontend + Backend API] in public_subnet
    service rds(database)[RDS MySQL] in private_subnet
    
    service s3(disk)[S3 Bucket\nAutomated Backups] in aws
    service cw(cloud)[CloudWatch\nMonitoring & Alerts] in aws
    
    igw:L -- R:ec2
    ec2:B -- T:rds
    ec2:R -- L:s3
    ec2:T -- B:cw
```

### Components
*   **Frontend (React/Vite)**: Hosted within the EC2 instance, served via Nginx.
*   **Backend API (Node.js/Express)**: Runs as a REST API on EC2.
*   **Database (MySQL)**: Hosted on Amazon RDS (Private Subnet) for security and automated backups.
*   **Storage (S3)**: Used for storing routine database dumps generated by our automation scripts.
*   **Monitoring (CloudWatch)**: Tracks CPU, Memory, and Disk utilization pushed via custom cron jobs.

## 2. VPC & Networking setup

A custom Virtual Private Cloud (VPC) ensures strict isolation between the application and database tiers.

*   **VPC CIDR**: `10.0.0.0/16`
*   **Public Subnet (`10.0.1.0/24`)**: Hosts the EC2 instance. It has an Internet Gateway attached to allow external web traffic (HTTP/HTTPS).
*   **Private Subnet (`10.0.2.0/24`)**: Hosts the RDS MySQL instance. No direct internet access.
*   **Security Groups**:
    *   **EC2 SG**: Allows Inbound HTTP (80), HTTPS (443), and SSH (22) from authorized IPs.
    *   **RDS SG**: Allows Inbound MySQL (3306) *only* from the EC2 SG.

## 3. Linux Administration & Automation

The EC2 instance runs **Ubuntu 22.04 LTS**. A series of bash scripts and system configurations were developed to automate provisioning and maintenance:

*   **Service Management (Systemd)**: The application runtime (Docker Compose) is registered as a `systemctl` service (`fanengage.service`) ensuring it automatically starts on boot and restarts on failure.
*   **Deployment Script**: `setup-server.sh` bootstraps the server by installing Docker, AWS CLI, and configuring `ufw` firewall rules.
*   **Monitoring Automation**: `monitor.sh` runs every 5 minutes via cron. It collects CPU, Memory, and Disk utilization using standard Linux tools (`top`, `free`, `df`) and pushes them to CloudWatch custom metrics.
*   **Backup Automation**: `backup-db.sh` runs daily at 2:00 AM via cron. It uses `mysqldump` to export the database, `gzip` to compress it, and `aws s3 cp` to upload it to the S3 bucket.

## 4. Docker Deployment

For standardized deployment across environments, the application uses Docker:
*   **Backend (`backend/Dockerfile`)**: A Node.js Alpine image exposing port 5001.
*   **Frontend (`frontend/Dockerfile`)**: A multi-stage build that compiles the React app and serves it using Nginx Alpine.
*   **Orchestration (`docker-compose.yml`)**: Allows running the frontend, backend, and a local MySQL instance with a single `docker-compose up -d` command.

## 5. Pricing Strategy & TCO Analysis

This section outlines the infrastructure pricing estimates across compute, storage, and networking, broken down by service tier, disaster recovery options, and regional expansions.

### Base Infrastructure Costs (Single Region - ap-south-1)
Assuming a production-grade initial setup (non-Free Tier):

| Service Category | Configuration & Usage Pattern | Estimated Monthly Cost |
| :--- | :--- | :--- |
| **Compute (EC2)** | 1x `t3.medium` (Linux), On-Demand | ~$30.36 |
| **Storage (EBS)** | 50GB General Purpose SSD (gp3) | ~$4.00 |
| **Database (RDS)** | 1x `db.t3.medium` (MySQL), Multi-AZ Disabled, 50GB | ~$55.00 |
| **Networking** | 100 GB Outbound Data Transfer | ~$9.00 |
| **Total Base Cost** | | **~$98.36 / month** |

### Monitoring & Analytics Tiers (CloudWatch)
*   **Standard Tier (Current)**: Basic infrastructure metrics (5-min intervals) + 3 custom scripts. (~$2.00/month)
*   **Premium SLA Tier**: 1-minute detailed monitoring, advanced application log analytics (CloudWatch Logs Insights), and automated incident response alarms. (~$15.00/month)

### Backup & Disaster Recovery (RPO/RTO aligned)
*   **Standard DR**: Daily Automated RDS Snapshots + S3 MySQL dumps (RPO: 24 hours, RTO: ~1 hour). Cost: ~$5.00/month for S3 Standard Storage (100GB).
*   **Enterprise DR**: Continuous RDS backups with Point-in-Time Recovery (PITR) enabled, and Cross-Region Snapshot replication (RPO: 5 minutes, RTO: ~15 minutes). Cost: ~$25.00/month.

### Multi-Region Deployment Costs (High Availability)
If expanding from `ap-south-1` (Mumbai) to include `ap-southeast-1` (Singapore) for redundancy:
*   **Active-Passive (Pilot Light)**: Replicating database to Singapore, but keeping compute instances scaled to zero until failover. (+ ~$60/month).
*   **Active-Active**: Running identical `t3.medium` web fleets behind Route 53 latency-based routing, with an RDS Read Replica. (+ ~$110/month).

### Infrastructure Optimization Recommendations (TCO Reduction)
1. **Compute Savings Plans**: Commit to a 1-year or 3-year Compute Savings Plan to reduce EC2 instance costs by up to 30-50% compared to On-Demand pricing.
2. **Auto Scaling**: Implement EC2 Auto Scaling Groups based on CloudWatch CPU metrics. Scale down to a single `t3.micro` instance during off-peak hours to save compute costs.
3. **Storage Lifecycle Policies**: Transition S3 database dumps older than 30 days to S3 Standard-IA or Glacier to reduce storage costs by up to 60%.

## 6. Application Dashboards

The developed web application directly addresses the business requirements:
*   **Admin/Manager Panel**: Role-based access controls for internal staff to monitor engagements and manage fans securely.
*   **Executive Analytics**: Interactive visual dashboards built using Recharts for ticket sales, revenue generation, and engagement trends.
*   **Monitoring & Alerting**: Real-time operational dashboards enabling proactive incident response.
*   **Operational Records**: Database-backed ledgers ensuring data integrity, auditability, and compliance for all fan events.
